系統運作架構說明

這是一個經典且無伺服器（Serverless）架構的「非同步任務處理」流程，包含了五個主要階段：

1 前端發起請求 (HTML / GitHub Pages)

使用者在畫面輸入網址後，JavaScript 將任務寫入 Firebase 的 tasks 集合（狀態 pending），並獲得專屬 taskId。前端隨即進入等待狀態，同時監聽回傳結果。

2 雲端中繼站 (Firebase)

擔任前後端溝通的唯一橋樑。它收到了前端貼上來的新任務佈告，默默等待後端來認領處理。

3 後端運算引擎 (Python Processor)

後端系統定時檢查 Firebase。一旦發現 pending 任務便標記為 processing，接著執行下載音檔 ➔ Whisper 轉錄 ➔ Gemini AI 總結分析的高強度運算。

4 雲端儲存 (Firebase Storage)

後端運算完成後，將生成的逐字稿 (.srt) 備份儲存到 Firebase Storage 之中，並產生專屬的下載連結。

5 結果回傳前端 (Firebase -> HTML)

後端將 AI 分析結果、Storage 下載連結連同原本的 taskId 寫回 Firebase。Firebase 察覺後立刻發送推播給等待中的前端 HTML，網頁隨即解除轉圈圈狀態這整份結果渲染至畫面。

1. YouTube 雲端禁令 (最沉痛的教訓)

在本地測試時 yt-dlp 運行得非常完美。但一旦部署至 Render 這種資料中心環境，YouTube 便會祭出最嚴厲的 IP 驗證，直接判定為機器人下載。在社群爬文與實驗無數種 Cookie 與 PO Token 解法後，考量到維護成本與雲端隱私安全，最終決定在雲端版暫時宣告放棄。

2. 突破 Groq 25MB 限制

Groq Whisper API 強制要求音檔在 25MB 以下。開發中花了大量時間研究 yt-dlp 參數，依據影片時長動態計算傳輸位元率，力求在不失真的情況下精準壓在 24.9MB 跳舞。

3. 雙重壓縮防線 (FFmpeg)

為了防止下載器失控，緊急追加了 FFmpeg 二次強制壓縮功能。雖然 Render 的免費 CPU 運算緩慢，但這是確保任務不被 API 彈回的最後一道保險。

4. Render 的「生存防護」技巧

為了省下每月 7 美金的背景服務費，我們使用了 Render Web Service 配合外部 Cron Job 工具。定時叫醒服務防止其休眠，這才達成了趨近 24 小時的待機運作。

5. 儲存與流量的現實考量

原本想提供 MP3 下載，但考量到 Firebase 免費版僅有 5GB 空間與 10GB 每月流量，幾部長篇 Podcast 就會塞爆。因此忍痛捨棄大檔下載，專注於輕量化的分析文本。

6. 與 Gemini API 的版本追逐戰

模型版本更新頻繁且不穩定。實驗過 latest-flash 塞車、2.5-flash 在雲端炸裂，最後選定 flash-lite-latest 測試是否能提高RPD每日用量。

7. 消失的秒數 (Groq 每小時限制)

Groq 的 Whisper 免費層級每小時僅提供 7,200 秒（120 分鐘）的轉錄額度。分析兩部長篇 Podcast 就會觸發熔斷，免費仔只能吞淚等下一個整點重置。
